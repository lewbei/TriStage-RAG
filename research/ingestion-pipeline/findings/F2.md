# F2: ColBERT indexing compression

## Findings

### [1] ColBERTv2 residual compression achieves 6-10x space reduction over ColBERTv1
- quote: "ColBERTv2 couples an aggressive residual compression mechanism with a denoised supervision strategy to simultaneously improve the quality and space footprint of late interaction... reducing the space footprint of late interaction models by 6-10x."
- url: https://arxiv.org/abs/2112.01488
- source_type: primary
- published: 2021-12-02 (v3: 2022-07-10)
- confidence: high

### [2] ColBERTv2 default indexing uses nbits=2, storing 2 bits per dimension of the residual vector
- quote: "config = ColBERTConfig(nbits=2, root=\"/path/to/experiments\")"
- url: https://github.com/stanford-futuredata/ColBERT/blob/main/colbert/indexing/collection_indexer.py
- source_type: primary
- published: unknown (code, main branch)
- confidence: high

### [3] Residual compression stores centroid ID (int32, 4 bytes) + quantized residual (nbits per dim, packed into uint8), yielding ~36 bytes per embedding for dim=128/nbits=2 vs 256 bytes uncompressed (float16)
- quote: "self.codes = codes.to(torch.int32)  # (num_embeddings,) int32" and "self.residuals = residuals  # (num_embeddings, compressed_dim) uint8" and "residuals_packed = residuals_packed.reshape(residuals.size(0), self.dim // 8 * self.nbits)"
- url: https://github.com/stanford-futuredata/ColBERT/blob/main/colbert/indexing/codecs/residual.py
- source_type: primary
- published: unknown (code)
- confidence: high

### [4] Residual quantization uses data-dependent quantile-based buckets computed from a 5% heldout sample (up to 50k embeddings), not uniform quantization
- quote: "num_options = 2 ** self.config.nbits" and "bucket_cutoffs = heldout_avg_residual.float().quantile(bucket_cutoffs_quantiles)" and "heldout_fraction = 0.05" and "heldout_size = int(min(heldout_fraction * sample.size(0), 50_000))"
- url: https://github.com/stanford-futuredata/ColBERT/blob/main/colbert/indexing/collection_indexer.py
- source_type: primary
- published: unknown (code)
- confidence: high

### [5] Number of centroids (partitions) is automatically computed as 2^floor(log2(16 * sqrt(num_embeddings_est))), scaling sublinearly with collection size
- quote: "self.num_partitions = int(2 ** np.floor(np.log2(16 * np.sqrt(self.num_embeddings_est))))"
- url: https://github.com/stanford-futuredata/ColBERT/blob/main/colbert/indexing/collection_indexer.py
- source_type: primary
- published: unknown (code)
- confidence: high

### [6] ColBERT index stores six file types per chunk: codes (int32 centroid IDs), residuals (uint8 packed quantized), doclens, metadata, plus global centroids.pt, avg_residual.pt, buckets.pt, inv.pid.pt, and metadata.json
- quote: "Saved data files described below: {CHUNK#}.codes.pt: centroid id for each embedding in chunk, {CHUNK#}.residuals.pt: 16-bits residual for each embedding in chunk, doclens.{CHUNK#}.pt: number of embeddings within each passage in chunk"
- url: https://github.com/stanford-futuredata/ColBERT/blob/main/colbert/indexing/collection_indexer.py
- source_type: primary
- published: unknown (code)
- confidence: high

### [7] PLAID engine reduces late interaction search latency by 7x on GPU and 45x on CPU vs vanilla ColBERTv2 using centroid interaction and centroid pruning, without impacting quality
- quote: "PLAID uses centroid interaction as well as centroid pruning... to reduce late interaction search latency by up to 7x on a GPU and 45x on a CPU against vanilla ColBERTv2, while continuing to deliver state-of-the-art retrieval quality."
- url: https://arxiv.org/abs/2205.09707
- source_type: primary
- published: 2022-05-19
- confidence: high

### [8] ColBERT index supports incremental updates (beta as of Jan 2023) and resume-based re-indexing via config.resume flag
- quote: "We have merged a new index updater feature and support for additional Hugging Face models! These are in beta so please give us feedback as you try them out." and in code: "if self.config.resume and self.saver.check_chunk_exists(chunk_idx): ... continue"
- url: https://github.com/stanford-futuredata/ColBERT/blob/main/colbert/indexing/collection_indexer.py
- source_type: primary
- published: 2023-01-29
- confidence: medium

### [9] ColBERTv2 embeds documents into 128-dimensional vectors (dim=128) by default, down from BERT's 768, which is a key contributor to the space savings
- quote: "config = ColBERTConfig(bsize=32, lr=1e-05, warmup=20_000, doc_maxlen=180, dim=128, ...)"
- url: https://github.com/stanford-futuredata/ColBERT/blob/main/README.md
- source_type: primary
- published: unknown (README)
- confidence: high

### [10] The binarization-only approach (1-bit compression) was explored but ColBERTv2's residual compression was found to be "more robust"
- quote: "binarization: ColBERT with a baseline binarization-based compression strategy (as opposed to ColBERTv2's residual compression, which we found to be more robust)."
- url: https://github.com/stanford-futuredata/ColBERT/blob/main/README.md
- source_type: primary
- published: unknown (README)
- confidence: high

## Suggested follow-ups
- Quantify exact storage per embedding at different nbits values (1, 2, 3, 4) to find the sweet spot for the 3.7GB → target reduction
- Investigate whether reducing dim from 128 to 64 or 96 (with model fine-tuning) would preserve retrieval quality at lower storage
- Explore whether the nbits=2 default can be reduced to nbits=1 (binarization) for the specific 10k-doc corpus without significant quality loss
