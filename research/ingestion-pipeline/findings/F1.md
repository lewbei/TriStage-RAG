# F1: SentenceTransformer batch encoding optimization

## Findings

### [1] Multi-GPU encoding via `device` list or `start_multi_process_pool()` gives near-linear speedup for large datasets
- quote: "You can encode input texts with more than one GPU (or with multiple processes on a CPU machine). It tends to help significantly with large datasets"
- url: https://www.sbert.net/examples/sentence_transformer/applications/computing-embeddings/README.html#multi-process-multi-gpu-encoding
- source_type: primary
- published: 2026
- confidence: high

### [2] The `chunk_size` parameter controls parallelism granularity independently of `batch_size`
- quote: "With a chunk_size=1000 and a batch_size=32, the input texts will be split into chunks of 1000 texts, and each chunk will be sent to a process and embedded in batches of 32 texts at a time. This can help with memory management and performance, especially for large datasets."
- url: https://www.sbert.net/examples/sentence_transformer/applications/computing-embeddings/README.html#multi-process-multi-gpu-encoding
- source_type: primary
- published: 2026
- confidence: high

### [3] Default batch_size=32; adjusting it can significantly improve speed, but optimal value requires per-hardware benchmarking
- quote: "Adjusting batch_size can significantly improve processing speed. The optimal value depends on your hardware, model size, precision, and input length. Benchmark a few batch sizes on a small subset of your data to find the best value."
- url: https://www.sbert.net/docs/package_reference/sentence_transformer/model.html#sentence_transformers.sentence_transformer.model.SentenceTransformer.encode
- source_type: primary
- published: 2026
- confidence: high

### [4] SentenceTransformer.encode() supports native fp16/int8/uint8/binary/ubinary quantization via the `precision` parameter
- quote: "precision (Literal['float32', 'int8', 'uint8', 'binary', 'ubinary'], optional) – The precision to use for the embeddings."
- url: https://www.sbert.net/docs/package_reference/sentence_transformer/model.html#sentence_transformers.sentence_transformer.model.SentenceTransformer.encode
- source_type: primary
- published: 2026
- confidence: high

### [5] SentenceTransformer supports three inference backends: torch, ONNX (with CUDAExecutionProvider), and OpenVINO
- quote: "backend (str, optional) – The backend to use for inference. Can be 'torch' (default), 'onnx', or 'openvino'."
- url: https://www.sbert.net/docs/package_reference/sentence_transformer/model.html#sentence_transformers.sentence_transformer.model.SentenceTransformer
- source_type: primary
- published: 2026
- confidence: high

### [6] Static embedding models achieve 100x-400x faster CPU inference than all-MiniLM-L6-v2 at 85%+ quality
- quote: "These models are 100x to 400x faster on CPU than common counterparts like all-mpnet-base-v2 and multilingual-e5-small, while reaching at least 85% of their performance on various benchmarks."
- url: https://huggingface.co/blog/static-embeddings
- source_type: primary
- published: 2025-01-15
- confidence: high

### [7] all-MiniLM-L6-v2 (22.7M params, 384-dim) encodes ~1739 sentences/sec on CPU and ~16942 sentences/sec on GPU
- quote: "sentence-transformers/all-MiniLM-L6-v2 ... CPU (sentences per second): 1739.31 ... GPU (sentences per second): 16942.46"
- url: https://huggingface.co/blog/static-embeddings
- source_type: primary
- published: 2025-01-15
- confidence: high

### [8] All-MiniLM-L6-v2 max_seq_length defaults to 256 tokens; reducing it can improve throughput for short documents
- quote: "Max Sequence Length: 256 ... model.max_seq_length = 200"
- url: https://www.sbert.net/examples/sentence_transformer/applications/computing-embeddings/README.html#input-sequence-length
- source_type: primary
- published: 2026
- confidence: high

### [9] Multi-GPU example with all-MiniLM-L6-v2 and 100k sentences demonstrates the official recommended pattern
- quote: "sentences = [f\"This is sentence {i}\" for i in range(100000)] ... pool = model.start_multi_process_pool() ... emb = model.encode(sentences, pool=pool)"
- url: https://github.com/huggingface/sentence-transformers/blob/main/examples/sentence_transformer/applications/computing-embeddings/computing_embeddings_multi_gpu.py
- source_type: primary
- published: 2026
- confidence: high

### [10] Matryoshka truncation reduces embedding dimension by 2x at only 1.47% quality loss, proportionally speeding up downstream tasks
- quote: "reducing the dimensionality by e.g. 2x only has a 1.47% reduction in performance (0.5031 NDCG@10 vs 0.4957 NDCG@10), while realistically resulting in a 2x speedup in retrieval speed."
- url: https://huggingface.co/blog/static-embeddings
- source_type: primary
- published: 2025-01-15
- confidence: high

### [11] torch.compile() integration is available for SentenceTransformer modules as a PyTorch 2.x optimization
- quote: "Compile this Module's forward using torch.compile(). This Module's __call__ method is compiled and all arguments are passed as-is to torch.compile()."
- url: https://www.sbert.net/docs/package_reference/sentence_transformer/model.html#sentence_transformers.sentence_transformer.model.SentenceTransformer
- source_type: primary
- published: 2026
- confidence: high

### [12] Large batch sizes improve contrastive training quality; all-MiniLM-L6-v2 was trained with batch_size=1024 on TPU v3-8
- quote: "We train the model during 100k steps using a batch size of 1024 (128 per TPU core)."
- url: https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2
- source_type: primary
- published: unknown
- confidence: high

## Suggested follow-ups
- Benchmark optimal batch_size for your specific GPU (e.g. RTX 3090 vs A100 vs T4) with your actual document lengths, since the optimal value is hardware-dependent.
- Evaluate ONNX Runtime export with CUDAExecutionProvider for your model to compare against native PyTorch inference speed.
- Quantify the throughput gain of multi-GPU encoding vs single-GPU with large batch_size for your 10k-100k document corpus.
