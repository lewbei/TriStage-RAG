# F4: FAISS indexing strategies

## Findings

### [1] Flat (brute-force) is optimal for small corpora (<10k docs) or low query counts (1k-10k queries)
- quote: "If you plan to perform only a few searches (say 1000-10000), the index building time will not be amortized by the search time. Then direct computation is the most efficient option."
- url: https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index
- source_type: primary
- published: 2026-03-26 (last edited)
- confidence: high

### [2] HNSW is the recommended default when memory is not a concern, offering the best speed/accuracy tradeoff
- quote: "If you have a lots of RAM or the dataset is small, HNSW is the best option, it is a very fast and accurate index. The 4 <= M <= 64 is the number of links per vector, higher is more accurate but uses more RAM."
- url: https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index
- source_type: primary
- published: 2026-03-26
- confidence: high

### [3] IVFFlat with K = 4*sqrt(N) to 16*sqrt(N) centroids is the standard for sub-1M datasets
- quote: "If below 1M vectors: ...,IVFK,... Where K is 4*sqrt(N) to 16*sqrt(N), with N the size of the dataset. This just clusters the vectors with k-means. You will need between 30*K and 256*K vectors for training (the more the better)."
- url: https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index
- source_type: primary
- published: 2026-03-26
- confidence: high

### [4] For 10k-100k documents, IVFFlat with nlist = 64-400 centroids (using 4*sqrt(N) rule) gives exact distances with occasional missed neighbors
- quote: "When the dataset is around 1m vectors, the exhaustive index becomes too slow, so a good alternative is IndexIVFFlat. It still returns exact distances but occasionally misses a neighbor because it is non-exhaustive."
- url: https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors
- source_type: primary
- published: 2022-01-11
- confidence: high

### [5] HNSW on SIFT1M (128D) achieves 98.87% recall@1 at 0.059ms/query with efSearch=128, dramatically outperforming IVFFlat
- quote: "HNSW obtains much better speed / precision operating points than IVFFlat (eg. 0.020 ms vs. 0.140 ms to get > 0.9 recall at 1), at a higher memory cost"
- url: https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors
- source_type: primary
- published: 2022-01-11
- confidence: high

### [6] HNSW memory cost is (d*4 + M*2*4) bytes per vector — roughly 3x the raw vector size for typical M=32
- quote: "The memory usage is (d * 4 + M * 2 * 4) bytes per vector."
- url: https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index
- source_type: primary
- published: 2026-03-26
- confidence: high

### [7] IVFFlat with nlist=16384 and 20 threads achieved 0.538s batch search on 1M vectors with R@1=0.8980, build time 240s
- quote: "IVF16384,Flat 0.538 s 0.8980 512 + 8 MB 240 s"
- url: https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors
- source_type: primary
- published: 2022-01-11
- confidence: high

### [8] PQ with FastScan (PQx4fs) offers the best accuracy-per-byte for memory-constrained regimes under 64 bytes/vector
- quote: "for the low-accuracy regime, the fast-scan PQ4 implementation offers the best tradeoff"
- url: https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors
- source_type: primary
- published: 2022-01-11
- confidence: high

### [9] For a 10k-doc RAG corpus with 768-dim embeddings, HNSW is strongly preferred over IVFFlat — it requires no training and scales trivially
- quote: "HNSW does only support sequential adds (not add_with_ids) so here again, prefix with IDMap if needed. HNSW does not require training and does not support removing vectors from the index."
- url: https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index
- source_type: primary
- published: 2026-03-26
- confidence: high

### [10] On 1M SIFT1M vectors, IVF+HNSW (nlist=4096, HNSW32) achieved 100% recall@100 at nprobe=146 with 916us search time and 523MB memory — vs. OPQ32,IMI2x8,PQ32 achieving 74% recall at 461us with only 41MB
- quote: "IVF4096_HNSW,Flat 90% 550us 523MB | OPQ32,IMI2x8,PQ32 74% 461us 41MB"
- url: https://www.pinecone.io/learn/series/faiss/composite-indexes/
- source_type: secondary
- published: unknown
- confidence: high

### [11] For a 100k-doc corpus, the nlist calculation yields ~1260-4000 centroids (4*sqrt(100k)=1265, 16*sqrt(100k)=5060), requiring 38k-1.01M training vectors
- quote: "You will need between 30*K and 256*K vectors for training (the more the better)."
- url: https://github.com/facebookresearch/faiss/wiki/Guidelines-to-choose-an-index
- source_type: primary
- published: 2026-03-26
- confidence: high

### [12] HNSW with scalar quantizer (HNSW,SQ) nearly doubles throughput vs. HNSW,Flat on SIFT1M with minor recall loss
- quote: "HNSW with a scalar quantizer is better than the classical HNSW (note that SIFT1M is originally encoded in bytes)"
- url: https://github.com/facebookresearch/faiss/wiki/Indexing-1M-vectors
- source_type: primary
- published: 2022-01-11
- confidence: high
