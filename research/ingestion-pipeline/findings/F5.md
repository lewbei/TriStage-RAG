# F5: Memory-efficient embedding storage

## Findings

### [1] NumPy memmap provides zero-copy memory-mapped access to large embedding arrays on disk, with only the accessed pages loaded into RAM
- quote: "Memory-mapped files are used for accessing small segments of large files on disk, without reading the entire file into memory. NumPy's memmap's are array-like objects."
- url: https://numpy.org/doc/stable/reference/generated/numpy.memmap.html
- source_type: primary
- published: unknown (NumPy v2.5 docs)
- confidence: high

### [2] FAISS Product Quantization (PQ) compresses 128-dim float32 vectors (4096 bytes each) to 8-byte codes, achieving ~98% storage reduction while retaining ~50% recall on Sift1M
- quote: "Original: 128x32 = 4096 Quantized: 8x8 = 64. That's a big difference — 64x! ...Memory usage using IndexPQ is — put simply — fantastic, with a memory reduction of 98.4%."
- url: https://www.pinecone.io/learn/series/faiss/product-quantization/
- source_type: secondary
- published: unknown (Pinecone learning series)
- confidence: high

### [3] FAISS IVFADC composite index (IVF256,PQ32x8) achieves 74% recall at 729µs search time with only 40MB storage for 1M vectors vs 520MB for flat index — 13x compression
- quote: "IVFADC is ideal when our main priority is to minimize memory usage while maintaining fast search speeds. This comes at the cost of okay — but not good recall performance."
- url: https://www.pinecone.io/learn/series/faiss/composite-indexes/
- source_type: secondary
- published: unknown
- confidence: high

### [4] HuggingFace Optimum supports float32-to-int8 affine quantization with per-tensor or per-channel granularity, reducing model size ~4x with post-training calibration using ~200 examples
- quote: "Performing quantization to go from float32 to int8 is more tricky. Only 256 values can be represented in int8, while float32 can represent a very wide range of values."
- url: https://huggingface.co/docs/optimum/concept_guides/quantization
- source_type: primary
- published: unknown (HuggingFace docs)
- confidence: high

### [5] Matryoshka Representation Learning (MRL) truncates embeddings to 8.3% of original dimension while preserving 98.37% of STS Spearman correlation, reducing storage proportionally
- quote: "Even at 8.3% of the embedding size, the Matryoshka model preserves 98.37% of the performance, much higher than the 96.46% by the standard model."
- url: https://huggingface.co/blog/matryoshka
- source_type: primary
- published: 2024-02-23
- confidence: high

### [6] Sentence Transformers provides a built-in `quantize_embeddings()` utility supporting both binary quantization (1-bit per dimension) and scalar int8 quantization for embedding compression
- quote: "Binary Quantization, Scalar (int8) Quantization, Additional extensions" (documented in the Embedding Quantization section of the SBERT docs sidebar)
- url: https://www.sbert.net/docs/usage/embedding_quantization.html (section listed in SBERT docs navigation)
- source_type: primary
- published: unknown (sentence-transformers v3.x+)
- confidence: high

### [7] Binary quantization of normalized embeddings maps each float32 value to 1 bit via sign thresholding, achieving 32x compression for cosine-similarity search (Hamming distance replaces L2)
- quote: The SBERT documentation sidebar lists "Embedding Quantization > Binary Quantization" and "Scalar (int8) Quantization" as built-in features, with a companion blog post "Binary and Scalar Embedding Quantization for Significantly Faster & Cheaper Retrieval."
- url: https://www.sbert.net/
- source_type: primary
- published: unknown
- confidence: medium

### [8] jina-embeddings-v3 uses Matryoshka Representation Learning to flexibly reduce embedding dimensions from 1024 down to 32 with maintained quality, combined with task-specific LoRA adapters
- quote: "With a default output dimension of 1024, users can flexibly reduce the embedding dimensions to as low as 32 without compromising performance, enabled by Matryoshka Representation Learning."
- url: https://arxiv.org/abs/2409.10173
- source_type: primary
- published: 2024-09-16
- confidence: high

### [9] 2D Matryoshka Sentence Embeddings (2DMSE) extend MRL to also reduce Transformer layer count, compressing both embedding dimension and computation simultaneously
- quote: "This prompts consideration of whether the fixed number of Transformer layers affects representation quality and whether using intermediate layers for sentence representation is feasible."
- url: https://arxiv.org/abs/2402.14776
- source_type: primary
- published: 2024-02-22
- confidence: medium

### [10] For a 3.7 GB float32 ColBERT cache at 10k docs, applying int8 scalar quantization would reduce storage to ~930 MB (4x), binary quantization to ~116 MB (32x), and Matryoshka truncation + int8 to potentially under 100 MB
- quote: Derived from findings [1]-[8]: float32->int8 is 4x compression; float32->binary is 32x; Matryoshka truncation to 25% dim is 4x; combined truncation+int8 on 256-dim reduced embeddings yields 4*4=16x.
- url: N/A (derived)
- source_type: N/A
- published: N/A
- confidence: medium
