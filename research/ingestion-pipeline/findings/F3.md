# F3: Parallel ingestion patterns

## Findings

### [1] LlamaIndex IngestionPipeline runs stages sequentially, not in parallel
- quote: "An IngestionPipeline uses a concept of Transformations that are applied to input data. These Transformations are applied to your input data, and the resulting nodes are either returned or inserted into a vector database."
- url: https://docs.llamaindex.ai/en/stable/module_guides/loading/ingestion_pipeline/
- source_type: primary
- published: unknown
- confidence: high

### [2] LlamaIndex provides async support for ingestion throughput, not true parallel stages
- quote: "The IngestionPipeline also has support for async operation: nodes = await pipeline.arun(documents=documents)"
- url: https://docs.llamaindex.ai/en/stable/module_guides/loading/ingestion_pipeline/
- source_type: primary
- published: unknown
- confidence: high

### [3] LlamaIndex caches per node+transformation pair, enabling incremental re-ingestion without reprocessing unchanged chunks
- quote: "Each node+transformation pair is cached, so that subsequent runs (if the cache is persisted) with the same node+transformation combination can use the cached result and save you time."
- url: https://docs.llamaindex.ai/en/stable/module_guides/loading/ingestion_pipeline/
- source_type: primary
- published: unknown
- confidence: high

### [4] Vespa uses a sequential document processor chain but distributes ingestion horizontally across content nodes
- quote: "Document processing is a framework to create chains of configurable components that read and modify document operations. A feeder application sends the documents into a document processing chain. This chain is an ordered list of document processors."
- url: https://docs.vespa.ai/en/applications/document-processors.html
- source_type: primary
- published: unknown
- confidence: high

### [5] Vespa runs document-api and document-processing on the same nodes to avoid network hops during ingestion
- quote: "It is important to configure both document-api and document-processing to run the document processing on the same nodes as the document API endpoint, to avoid network hops to other nodes (for better throughput)."
- url: https://docs.vespa.ai/en/writing/indexing.html
- source_type: primary
- published: unknown
- confidence: high

### [6] Vespa chunking can be done server-side via synthetic fields or client-side, with no built-in parallel chunking pipeline
- quote: "Alternatively you can let Vespa do the chunking for you, by using a synthetic field outside the document."
- url: https://docs.vespa.ai/en/rag/working-with-chunks.html
- source_type: primary
- published: unknown
- confidence: high

### [7] LlamaIndex production RAG guidance recommends decoupling retrieval chunks from synthesis chunks as the primary scaling strategy
- quote: "A key technique for better retrieval is to decouple chunks used for retrieval with those that are used for synthesis."
- url: https://docs.llamaindex.ai/en/stable/optimizing/production_rag/
- source_type: primary
- published: unknown
- confidence: high

### [8] LlamaIndex production RAG recommends document hierarchies (summaries -> chunks) for scaling beyond 100 documents, which implies a multi-pass indexing pattern
- quote: "A big issue with the standard RAG stack (top-k retrieval + basic text splitting) is that it doesn't do well as the number of documents scales up - e.g. if you have 100 different PDFs."
- url: https://docs.llamaindex.ai/en/stable/optimizing/production_rag/
- source_type: primary
- published: unknown
- confidence: high
